## Question 1
**Asked by**: Dunkaroo Destroyer

**Question**: This requirement targets a recurring failure mode — agents littering project files with content that does not belong (diary-like updates, "what I just did" narrative, completed-work summaries), with PROJECT-STATUS.md as the worst offender. Before we design any mechanism, I want to pin the scope of files.

Is this requirement scoped specifically to PROJECT-STATUS.md, or to all living GRIPS documents (requirements.md, design.md, ADRs, Q&A sessions)?

My suggestion: make PROJECT-STATUS.md the primary target of the concrete rule, because it is the litter magnet — it is the one file whose whole job is "freeform current state," and that open-prose remit is exactly what invites the diary urge. But state the *underlying principle* generally ("every piece of content has exactly one correct home") so it naturally governs the other files too, without us having to write a separate rule per file.

**Answer**: Agreed. Scope the concrete rule to PROJECT-STATUS.md as the primary target (it is the litter magnet), and state the underlying principle generally — "every piece of content has exactly one correct home" — so it governs all living GRIPS documents without a separate per-file rule.

## Question 2
**Asked by**: Dunkaroo Destroyer

**Question**: If the organizing principle is "every piece of content has exactly one correct home," then the central design artifact is the routing table — the map of content-type to home. Most kinds route cleanly to homes GRIPS already has: "what changed" belongs in the git commit message; a durable decision or gotcha belongs in an ADR; a requirement decision belongs in a Q&A session; "where we are / what's next" belongs in PROJECT-STATUS.

The hard case is the diary/narrative urge itself — feelings, process reflection, "here's how the session went" — which has real value to some workflows (some agents are even explicitly instructed to journal) but no place in a spec tree. A pure prohibition gives that urge nowhere to go, so it leaks into status. Two ways to handle it:

- Option A — Sanction a journal home: GRIPS blesses a specific location for diary/narrative content (e.g., a gitignored NOTES file or a `.grips/journal/` directory) so the urge has somewhere legitimate to land, away from the spec tree.
- Option B — Rule it out of GRIPS entirely: GRIPS-managed files are diary-free by definition; the methodology names the alternative homes (git commit messages for factual "what changed"; the agent's own tool-level journaling, outside the managed surface) but does not itself host a diary.

Which way for the diary/narrative urge — sanction a home inside the project, or route it out of the GRIPS-managed surface entirely?

My suggestion: Option B, with one nuance. GRIPS should not host a diary — the moment we bless a journal *inside* the managed tree, it becomes a new litter magnet and people start mining it as if it were spec. But the methodology should explicitly *acknowledge* the urge and name where it goes instead, rather than only forbidding it, because an unnamed alternative is why the urge leaks today. A gitignored scratch file is fine as "explicitly not GRIPS's concern," but it stays outside the methodology's managed surface — GRIPS neither requires nor reads it.

**Answer**: Option B, confirmed. GRIPS does not sanction an in-tree journal — the diary urge is minor in practice (the audit found it essentially absent; commit messages already absorb narrative). The requirement is reframed around the audit's findings: a four-destination routing table, suppression of the completed-work changelog vessel, and pointer-discipline for issue references. Endorsed to proceed to concrete mechanism design.

## Audit: PROJECT-STATUS.md litter across three real projects (evidence for Q2)

Before deciding Q2, the histories of PROJECT-STATUS.md were audited in three GRIPS-driven projects to see what actually accumulates, and whether GitHub issues (adopted mid-flight) absorb any of it.

**Projects examined**:
- margo-md — 11 commits on the file; uses GitHub issues heavily (23). Peaked ~120 lines, manually cut to ~43.
- vocabreader — 29 commits on docs/PROJECT-STATUS.md; tracked bugs in an in-repo docs/ISSUES.md (not GitHub). Grew 136 → 467 lines, panic-cut to 156, immediately regrew.
- pmus — 31 commits on the file; adopted GitHub issues only at the finish line (research items #1–#6). "Recent Activity" grew to ~13 milestone bullets before a manual prune.

**The dominant litter is NOT the diary/feelings urge.** Across all three, emotional/reflective diary content was essentially absent. The real litter, in order of prevalence, was:

1. **Completed-work changelog** — a hand-maintained "Recent Activity" / "Completed Milestones" section duplicating git history. This was the single biggest litter source in every project. In all three it lived under a standing, invitational heading that kept regenerating the changelog after each manual purge.
2. **Outstanding-work duplication** — incomplete items restated in status ("What's Needed / Why / Context") that were, or should have been, tracked elsewhere. In pmus the same six research items were triplicated across the brief, requirements.md, and status.
3. **Durable technical findings / gotchas** — e.g. an SQLite `mode=ro` WAL-staleness finding (pmus), an xcodebuild/devicectl device-ID gotcha (margo-md), an HTTPServer/Readium resolution (vocabreader). ADR material, parked in status.
4. **Process/session narrative** — "what the human said this session," self-corrections, environment asides. Present but low-volume; the heavier persona/narrative voice lived in commit messages, which is its healthy outlet.

**Two cross-project patterns**:

- **Bloat → panic manual cut → immediate regrowth.** Every project shows this cycle. One vocabreader commit openly diagnoses it: "Rewrite PROJECT-STATUS.md ... 467 lines → 156 lines ... eliminates triple-tracking." Manual purges do not hold — proving SUPPRESS-by-exhortation alone fails, and the fix must be structural.
- **The "Recent Activity" heading is the shared villain.** A permanent invitational header is what keeps regenerating the changelog. Removing the vessel is the highest-leverage single change.

**GitHub issues finding (the specific sub-question)**: Adopting issues demonstrably shrank the outstanding-work litter — margo-md's f0982aa collapsed a nine-item UX backlog to one line per issue; pmus's d8b9242 collapsed six triplicated research items to a single pointer. So issues ARE the right home for incomplete work (they carry status natively; status shrinks to a pointer). Two caveats: issues address only litter type 2 (outstanding work), not the changelog (type 1) or findings (type 3); and discipline regressed in margo-md, where HEAD re-inlined issue bodies back into status — so issues require an explicit pointer-discipline rule (reference issue numbers, never restate them).

**Unanimous verdict from all three audits**: REDIRECT is the primary lever, paired with a targeted SUPPRESS of the container (the "Recent Activity" section), because every litter category already has a natural home the projects actively use — git log / commit messages (changelog), GitHub issues (outstanding work), ADRs (durable findings). QUARANTINE (a walled-off dumping section) was rejected by all three as merely a "sanctioned litter box" that legitimizes the accretion.

**Implication for Q2**: the diary-home question (sanction vs route-out) turns out to target a minor problem — the diary urge is small and commit messages already absorb it, favoring Option B. The larger requirement reframes around a routing table with four destinations plus suppression of the changelog vessel.

## Question 3
**Asked by**: Dunkaroo Destroyer

**Question**: The audit showed manual purges do not hold — the fix must be structural. The structural backbone is a *closed template* for PROJECT-STATUS.md: a fixed set of permitted sections, plus one meta-rule — if a piece of content does not fill one of these sections, it does not belong in the file (route it out per the routing table). A closed schema resists litter the way an open-prose remit invites it.

Proposed permitted sections:
1. Header — Current Phase, Current Stage, Active Work (1–2 sentences, forward-looking only).
2. Layer Synchronization — which layers are updated / in-sync / pending.
3. Outstanding Work — incomplete items only, expressed as issue pointers (issue number + one line) where a tracker exists; inline resume-context reserved for work too granular or ephemeral to warrant an issue.
4. Next Steps — short ordered list of what comes next (incomplete items only).
5. Sessions & Gates — index of Q&A/ADR sessions with their status, and approval gates passed. This is state, not narrative: session names and statuses, never accomplishment bullets. This section replaces the old "Recent Activity" heading, which the audit identified as the changelog-litter vessel.

Meta-rule: these are the only permitted sections. No "Recent Activity" / changelog / "what was accomplished" section under any name. Content that does not fit a permitted section is routed to its home, not written here.

Does this field set define the closed template, and specifically: is replacing "Recent Activity" with a strictly-scoped "Sessions & Gates" (state only) the right way to preserve the legitimate session/gate index while killing the changelog it was breeding?

My suggestion: adopt this field set. The critical move is the fifth section — the old "Recent Activity" heading conflated two things: a legitimate, forward-looking *index of sessions and gates* (which is allowed) and an illegitimate *accomplishment changelog* (which is the litter). Splitting them by renaming the section to "Sessions & Gates" and defining it as state-only keeps the useful half and removes the invitation for the harmful half. Pair that with the meta-rule so the closed schema is enforceable rather than aspirational.
