## Question 1
**Asked by**: Dunkaroo Destroyer

**Question**: This requirement targets a recurring failure mode — agents littering project files with content that does not belong (diary-like updates, "what I just did" narrative, completed-work summaries), with PROJECT-STATUS.md as the worst offender. Before we design any mechanism, I want to pin the scope of files.

Is this requirement scoped specifically to PROJECT-STATUS.md, or to all living GRIPS documents (requirements.md, design.md, ADRs, Q&A sessions)?

My suggestion: make PROJECT-STATUS.md the primary target of the concrete rule, because it is the litter magnet — it is the one file whose whole job is "freeform current state," and that open-prose remit is exactly what invites the diary urge. But state the *underlying principle* generally ("every piece of content has exactly one correct home") so it naturally governs the other files too, without us having to write a separate rule per file.

**Answer**: Agreed. Scope the concrete rule to PROJECT-STATUS.md as the primary target (it is the litter magnet), and state the underlying principle generally — "every piece of content has exactly one correct home" — so it governs all living GRIPS documents without a separate per-file rule.

## Question 2
**Asked by**: Dunkaroo Destroyer

**Question**: If the organizing principle is "every piece of content has exactly one correct home," then the central design artifact is the routing table — the map of content-type to home. Most kinds route cleanly to homes GRIPS already has: "what changed" belongs in the git commit message; a durable decision or gotcha belongs in an ADR; a requirement decision belongs in a Q&A session; "where we are / what's next" belongs in PROJECT-STATUS.

The hard case is the diary/narrative urge itself — feelings, process reflection, "here's how the session went" — which has real value to some workflows (some agents are even explicitly instructed to journal) but no place in a spec tree. A pure prohibition gives that urge nowhere to go, so it leaks into status. Two ways to handle it:

- Option A — Sanction a journal home: GRIPS blesses a specific location for diary/narrative content (e.g., a gitignored NOTES file or a `.grips/journal/` directory) so the urge has somewhere legitimate to land, away from the spec tree.
- Option B — Rule it out of GRIPS entirely: GRIPS-managed files are diary-free by definition; the methodology names the alternative homes (git commit messages for factual "what changed"; the agent's own tool-level journaling, outside the managed surface) but does not itself host a diary.

Which way for the diary/narrative urge — sanction a home inside the project, or route it out of the GRIPS-managed surface entirely?

My suggestion: Option B, with one nuance. GRIPS should not host a diary — the moment we bless a journal *inside* the managed tree, it becomes a new litter magnet and people start mining it as if it were spec. But the methodology should explicitly *acknowledge* the urge and name where it goes instead, rather than only forbidding it, because an unnamed alternative is why the urge leaks today. A gitignored scratch file is fine as "explicitly not GRIPS's concern," but it stays outside the methodology's managed surface — GRIPS neither requires nor reads it.
