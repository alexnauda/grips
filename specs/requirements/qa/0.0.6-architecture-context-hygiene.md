# GRIPS Requirements Q&A Session 0.0.6

**Phase**: 0.0 (Prototype)
**Date Started**: 2026-07-09
**Participants**: Dunkaroo Destroyer (Agent), Floppy Dog (Human)
**Purpose**: Decide GRIPS's stance on harness and model coupling — whether to adopt a two-layer architecture (a portable methodology core plus an optional, separated adapter layer such as Claude Code skills or hooks) — and open the "context hygiene" theme, including delegating heavy work to sub-agents to keep human-with-agent sessions clean

---

## Background

This session arises from two connected questions raised during Q&A 0.0.5:

1. Whether GRIPS should introduce harness features (Claude Code skills or hooks) to enforce or operationalize the methodology — for example, a skill that runs the closed status template, or a hook that rejects a forbidden heading. The 0.0.5 audit established that exhortation-class fixes regress (manual purges did not hold), which makes deterministic enforcement attractive but raises the risk of coupling GRIPS to one harness.

2. A broader idea: preferring to perform implementation — and possibly research and drafting — in sub-agents as much as possible, to keep the context of the human-with-agent session clean. This connects directly to the 0.0.5 litter finding: much status/diary litter is the session straining to record what it did; if heavy work happens in a sub-agent that reports back a clean summary, the main session and status file stay clean by construction.

The governing constraint stated by Floppy Dog: keep GRIPS as model- and harness-agnostic as possible.

## Question 1
**Asked by**: Dunkaroo Destroyer

**Question**: Should GRIPS formally adopt a two-layer architecture as a core principle?

- Layer 1 — the methodology: portable, harness- and model-agnostic prose. It is the single source of truth and must be fully executable by any capable human-plus-agent pair with zero harness features.
- Layer 2 — adapters (optional, one per harness): thin, swappable operationalizations of Layer 1 (e.g., a Claude Code skill or hook bundle, a Cursor rules file). An adapter implements Layer 1; Layer 1 never depends on an adapter. They live in a clearly separated location (a dedicated directory or a sibling repository), never interleaved with the methodology text.

The defining invariant: strip every adapter and the methodology still works — just more manually. Adapters are accelerators, not requirements.

Does GRIPS adopt this two-layer model as an explicit principle?

My suggestion: yes, ratify it as a core principle now — but build no adapter yet. It directly resolves the tension between avoiding harness lock-in and wanting automation's benefits: agnosticism is preserved precisely because adapters are optional accelerators. It also gives future enforcement a home — the diary rule from 0.0.5 could gain a deterministic hook in Layer 2 — without contaminating the portable core. Ratifying the principle is cheap and shapes every later decision (including how sub-agent delegation and agent project management get expressed); building adapters is premature until the underlying rules are stable and a cluster of mechanisms justifies the coupling.

**Answer**: Yes. GRIPS adopts the two-layer architecture as a core principle — a portable, harness- and model-agnostic methodology (Layer 1) that is the single source of truth and fully executable with zero harness features, plus an optional, clearly-separated adapter layer (Layer 2) that operationalizes it per harness. The invariant holds: strip every adapter and the methodology still works. No adapter is built yet; the principle is ratified now to govern later decisions.

## Question 2
**Asked by**: Dunkaroo Destroyer

**Question**: Should GRIPS adopt a separation between a control channel and execution channels as a methodology principle, to keep the durable human-with-agent session clean?

The distinction:
- Control channel — the main human-with-agent session: decisions, Q&A, approvals, status maintenance, and spec propagation. This is the durable, context-sensitive conversation whose cleanliness matters most.
- Execution channels — bounded work that produces a discrete deliverable: implementation of a plan step, research, and (case-by-case) drafting. This work generates large volumes of transcript noise that has no lasting value once the deliverable exists.

The principle: isolate execution from the control channel so its context noise never pollutes the durable session — delegate it to a sub-agent where the harness supports one, or to a separate session otherwise. Only clean results flow back into the control channel: the deliverable itself, a concise summary, and any decisions that must land in Q&A or an ADR — never the raw execution transcript.

This connects directly to Q&A 0.0.5: much status and diary litter is the session straining to record what it did. If execution happens in an isolated channel, its "what I did" narrative dies with that channel's context, and the control channel and PROJECT-STATUS.md stay clean by construction — a structural fix, stronger than any prohibition.

Framed this way, is it a portable Layer-1 principle? And which work types are in scope by default — implementation only, or also research and drafting?

My suggestion: adopt it, and frame it in terms of channels and roles rather than any specific harness's sub-agent feature, so it stays Layer 1 (a harness with sub-agents satisfies it one way; a human spawning a separate session satisfies it another). On scope: recommend — not mandate — delegating implementation and research by default, since they are the largest context polluters and each produces a discrete deliverable that summarizes cleanly. Treat drafting as case-by-case: drafting spec prose often benefits from the control channel's accumulated context and tight human back-and-forth, so it is a weaker fit for isolation. Define precisely what returns to the control channel (deliverable, summary, decisions-for-Q&A/ADR) so delegation reduces noise without losing signal.

**Answer**: Adopted as a principle, framed as control vs. execution channels (Layer 1, harness-agnostic). Scope per suggestion: recommend (not mandate) delegating implementation and research by default; treat drafting case-by-case. What returns to the control channel is bounded to the deliverable, a concise summary, and any decisions destined for Q&A or an ADR — never the raw execution transcript.

## Current Understanding (design decisions for methodology integration)

Q&A 0.0.6 adds two requirements-layer principles to the methodology:

1. **Two-layer architecture (harness/model agnosticism)** — GRIPS is a portable methodology (Layer 1) that is the single source of truth and fully executable with zero harness features, plus an optional, clearly-separated adapter layer (Layer 2) that operationalizes it per harness (e.g., a Claude Code skill or hook). Adapters implement Layer 1; Layer 1 never depends on them. Invariant: strip every adapter and the methodology still works. No adapter is built in this phase.

2. **Control channel vs. execution channels (context hygiene)** — separate the durable human-with-agent session (decisions, Q&A, approvals, status, propagation) from bounded execution work (implementation, research, and case-by-case drafting). Isolate execution so its transcript noise never pollutes the control channel — a sub-agent where available, a separate session otherwise. Only the deliverable, a concise summary, and decisions-for-Q&A/ADR return; never the raw transcript. This is the structural form of the 0.0.5 anti-litter goal: an isolated execution channel's "what I did" narrative dies with its context, keeping the control channel and PROJECT-STATUS.md clean by construction.

These two principles, together with the 0.0.5 closed status template, form a coherent **context-hygiene** theme. The parked agent-project-management work (Q&A 0.0.4) is related but out of scope here.
